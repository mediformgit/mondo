# /improve-lesson — 既存講義を鑑定・改善する

質を上げるが、**全講を同じ形に寄せない**。これは厳守。**推奨モデル: Opus 4.8 / effort 高め**。
四半期レビューや、解析で離脱が出た講のメンテにも使える。

## 最初に必ず読む
1. [PRINCIPLES.md](../../../PRINCIPLES.md) — 不可侵の原則。
2. [docs/QUALITY-RUBRIC.md](../../../docs/QUALITY-RUBRIC.md) — **特に §8 既存講義の改善（鑑定）基準**。大原則＝「欠陥を直す。均一化はしない」。
3. [lib/lessons.ts](../../../lib/lessons.ts) — 対象の講義と、周辺講義（独自性の比較のため）。

## モードA：監査（引数なし）
1. 全講を §8 の8観点で健康診断（特に①正確さ・鮮度はWebで要確認）。
2. **編集はしない。** 優先度つきの所見リストを出す：各講ごとに「欠陥（観点）／深刻度／推奨アクション／独自性は保てるか」。
3. 似てきた講義の組があれば「**統一でなく差別化**」の方針で指摘。
4. 運営者に渡す（どれを `/improve-lesson <slug>` で直すか選べるように）。

## モードB：修正（slug指定）
1. **その講の“意図”を言語化** — 何を、誰に、どんな角度で教える講かを先に明文化（以後これを守る）。
2. **健康診断（§8）** — 欠陥だけを特定。流動的事実はWeb検証。検証不能なら**保留→報告して終了**。
3. **最小差分で改善** — 壊れている所だけ直す。著者の構成・声・例は、それ自体が欠陥でない限り**保つ**。
   - ✅ 直す：事実誤り/古さ、リンク切れ、論理の飛躍、未説明の用語、水増し/slop、目標未達、アクセシビリティ、`try`の死んだリンク。
   - ❌ やらない：共通テンプレへの寄せ、長さ/構成の正規化、他講に似せる変更。
   - ⚖️ **「最小差分」＝萎縮ではない。** その講の意図に資する欠陥なら、**構造的な修正も恐れず行う**（例：論理の飛躍を埋めるための段落追加）。やらないのは“統一のための”変更だけ。本物の欠陥を「いじると均一化に見えるかも」と放置しない。
   - 🔒 **`slug` と `num` は変えない**（ページURL・sitemap・前後ナビの順序が壊れる）。変更が必要に思えたら運営者に相談。
4. **独自性ガード（反・均一化）** — 提案する各変更が「より一般的」「他講義に似た方向」なら**却下**。迷ったら独自性を**研ぐ**側に倒す。
5. **ビルド** — `lib/lessons.ts` を更新し **`npm run build` を通す**。
6. **PR** — 作業ツリーがクリーンで `main` が最新であること（`git switch main && git pull`）を確認 → `git checkout -b improve/<slug>` → コミット → PR（`gh`認証済みなら `gh pr create`、無ければブランチまで報告）。PR本文に：
   - その講の「意図」、**直した欠陥（観点ごと）**、各変更が**どの欠陥に対応するか**、§4の添付（不確実な点／出典）。
   - 「他と揃えるため」を理由にした変更は**書かない・しない**。

## 守る不変条件
- 1講ずつ・差分・人の承認。**自動 rewrite / 自動 merge 禁止。** コンテンツは `lib/*.ts` に閉じる。
- 既定モデル `claude-opus-4-8`・adaptive thinking のみ。CSP/無料/無登録/広告なしを崩さない。

> 厳しさは「テンプレへの統一」ではなく「欠陥の除去」。多様性は守るべき品質である。
